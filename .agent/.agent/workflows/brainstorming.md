# 头脑风暴：将想法转化为设计

## 第一性原理

**为什么要头脑风暴？** 代码是思想的具体化。如果思想模糊，代码必然混乱。先厘清"做什么"和"为什么做"，再考虑"怎么做"。

**核心公理：**
1. 理解 > 行动：不理解需求就动手 = 浪费时间
2. 问题定义决定解决方案质量
3. 好的设计来自迭代对话，不是一次拍板

## 流程

### 1. 理解想法
- 先检查当前项目状态（文件、文档、最近提交）
- **一次只问一个问题**，避免信息过载
- 优先使用选择题，降低回答负担
- 聚焦于：目的、约束、成功标准

### 2. 探索方案
- 提出 2-3 个不同方案及其权衡
- 给出你的推荐并说明理由
- 让用户做选择，不要替用户决定

### 3. 呈现设计
- 理解清楚后，分段呈现设计（每段 200-300 字）
- 每段之后确认："这部分看起来对吗？"
- 覆盖：架构、组件、数据流、错误处理、测试

### 4. 输出文档
- 将验证过的设计写入 `docs/plans/YYYY-MM-DD-<主题>-design.md`
- 提交到 git

## 关键原则

- **一次一个问题** - 不要用问题轰炸用户
- **YAGNI** - 你不需要的功能，不要加
- **渐进验证** - 分段确认，随时调整
