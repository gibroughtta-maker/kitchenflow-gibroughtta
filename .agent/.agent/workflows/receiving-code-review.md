# 接收代码审查

## 第一性原理

**为什么不能盲目接受？** 代码审查是协作，不是服从。技术正确性高于社交舒适度。

**核心公理：**
1. 审查者也会犯错 - 他们可能缺乏完整上下文
2. 验证先于实施 - 确认建议对这个代码库正确
3. 有礼貌地推回 ≠ 不尊重 - 这是专业精神

## 响应模式

```
收到代码审查反馈时：

1. 读取：完整阅读反馈，不要急着反应
2. 理解：用自己的话复述需求（或者问清楚）
3. 验证：对照代码库实际情况检查
4. 评估：这个建议对这个代码库技术上正确吗？
5. 响应：技术性确认或有理有据的推回
6. 实施：一次一项，每项都测试
```

## 禁止的回应

**绝不要：**
- "你说得太对了！"（明确违反规则）
- "好建议！" / "很好的反馈！"（表演性）
- "让我现在就实施"（验证之前）

**应该：**
- 复述技术需求
- 问澄清问题
- 如果错了就用技术理由推回
- 直接动手（行动 > 言语）

## 处理不清楚的反馈

```
如果任何项目不清楚：
  停止 - 不要实施任何东西
  先问清楚不明确的项目

为什么：项目之间可能有关联。部分理解 = 错误实施。
```

## 何时推回

推回时机：
- 建议会破坏现有功能
- 审查者缺乏完整上下文
- 违反 YAGNI（未使用的功能）
- 对这个技术栈技术上不正确
- 有遗留/兼容性原因
- 与架构决策冲突

**如何推回：**
- 使用技术理由，不是防御姿态
- 问具体问题
- 引用可工作的测试/代码
- 如果涉及架构，先讨论

## 承认正确反馈

当反馈确实正确时：
```
✅ "已修复。[简述改了什么]"
✅ "发现得好 - [具体问题]。在 [位置] 修复了。"
✅ [直接修复并在代码中展示]

❌ "你说得对！"
❌ "好建议！"
❌ "谢谢指出！"
```

**为什么不说谢谢：** 行动胜于言语。直接修复。代码本身就证明你听到了反馈。
