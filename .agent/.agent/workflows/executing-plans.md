# 执行计划

## 第一性原理

**为什么要分批执行？** 人类的注意力有限，一次审查太多内容会遗漏问题。小批次+检查点 = 持续对齐。

**核心公理：**
1. 计划是假设，执行是验证 - 执行过程会发现计划的漏洞
2. 及时反馈减少返工 - 问题越早发现，修复成本越低
3. 书面计划 > 口头约定 - 白纸黑字减少歧义

## 流程

### 步骤 1：加载和审查计划
1. 读取计划文件
2. 批判性审查 - 识别任何问题或疑虑
3. 如有疑虑：开始前与人类伙伴讨论
4. 如无疑虑：创建待办列表并继续

### 步骤 2：执行批次
**默认：前 3 个任务**

对每个任务：
1. 标记为"进行中"
2. 严格按计划步骤执行
3. 运行指定的验证
4. 标记为"已完成"

### 步骤 3：报告
批次完成后：
- 展示实现了什么
- 展示验证输出
- 说："准备好接收反馈了。"

### 步骤 4：继续
根据反馈：
- 必要时应用更改
- 执行下一批次
- 重复直到完成

### 步骤 5：完成开发
所有任务完成并验证后：
- 使用 `finishing-a-development-branch` 工作流完成

## 何时停下来求助

**立即停止执行：**
- 批次中途遇到阻塞（缺少依赖、测试失败、指令不清）
- 计划有关键缺口导致无法开始
- 你不理解某个指令
- 验证反复失败

**宁可问清楚，不要瞎猜。**
